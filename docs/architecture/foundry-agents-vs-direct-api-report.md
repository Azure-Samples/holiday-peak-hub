# Technical Report: What We Lose by Switching from Foundry Agents to Direct Model Calls (Responses API)

<!-- Last Updated: 2026-04-30 -->

**Date**: 2026-04-28  
**Status**: Decision Input (Rejected ‚Äî Foundry-only policy adopted per ADR-005)  
**Author**: Architecture Review (automated analysis)  
**Scope**: holiday-peak-hub ‚Äî 26 retail agent services on AKS

---

## Executive Summary

Switching from Foundry Agents (via the Microsoft Agent Framework SDK) to direct Responses API / Chat Completions calls would eliminate **2‚Äì5 seconds of per-request overhead** on top of model inference time. For latency-sensitive retail paths (cart intelligence, order status, catalog search), this is a material improvement that directly impacts user experience.

However, the Foundry Agents layer provides a significant amount of managed infrastructure that our codebase actively depends on. Our current integration in [foundry.py](../../lib/src/holiday_peak_lib/agents/foundry.py) uses **server-side session continuity** (`AgentSession` with `service_session_id`), **automatic agent versioning with instruction drift detection** (`create_version` + `_check_instruction_drift`), **MAF middleware pipeline** (`FunctionInvocationLayer` for callable tools, `ChatOptions` for runtime parameters), and **deeply integrated OpenTelemetry instrumentation** (`AIProjectInstrumentor` + `AIInferenceInstrumentor` flowing to Application Insights). Removing the Foundry Agents layer means rebuilding or forfeiting each of these capabilities.

The recommended path is a **hybrid architecture**: migrate latency-critical, stateless agent paths (catalog search, order status, product enrichment) to direct Responses API while retaining Foundry Agents for complex, multi-turn, tool-heavy workflows (cart intelligence, support assistance, campaign intelligence). This preserves the latency win where it matters most while avoiding the cost of reimplementing session management, versioning, and tool orchestration from scratch.

## 1. Telemetry & Observability

### What We Currently Have

Our telemetry stack is initialized in [telemetry.py](../../lib/src/holiday_peak_lib/utils/telemetry.py) (`FoundryTracer._initialize_foundry_instrumentation`):

1. **Azure Monitor** ‚Äî `configure_azure_monitor(connection_string=...)` sets up the OpenTelemetry exporter to Application Insights.
2. **AIProjectInstrumentor** ‚Äî `AIProjectInstrumentor().instrument()` from `azure.ai.projects.telemetry` auto-instruments all `AIProjectClient` calls, including agent invocations.
3. **AIInferenceInstrumentor** ‚Äî `AIInferenceInstrumentor().instrument()` from `azure.ai.inference.tracing` auto-instruments direct inference calls.

Additionally, our `FoundryTracer` class maintains an in-memory event store (`_events` deque) with structured trace records for model invocations, tool calls, decisions, and evaluations ‚Äî all correlated via OpenTelemetry trace IDs and our custom `correlation_id`.

### What We Lose

| Telemetry Capability | Foundry Agents | Direct API | Impact |
|---|---|---|---|
| **Agent-level OTel spans** | Auto-generated by MAF SDK: `foundry.agent.run`, `foundry.agent.session`, `foundry.agent.tool` | Not generated ‚Äî only raw HTTP spans from `azure.ai.projects` client | **HIGH** ‚Äî Lose hierarchical span tree showing agent ‚Üí session ‚Üí tool ‚Üí model call chain |
| **Token usage breakdown** | `response.usage_details` includes `input_tokens`, `output_tokens`, `reasoning_tokens`, `cached_tokens` via MAF response | Available on `responses.create()` response, but format differs slightly | **LOW** ‚Äî Token data is available on both paths; minor parsing adjustment needed |
| **Foundry portal "Tracing" blade** | Populated by `AIProjectInstrumentor` spans correlated with agent name/session | Still populated if `AIProjectInstrumentor` is active ‚Äî traces direct API calls too | **MEDIUM** ‚Äî Traces still appear but without agent-level semantic grouping |
| **Foundry portal agent dashboard** | Shows invocations, latency percentiles, error rates per agent name | **Lost entirely** ‚Äî dashboard requires Foundry Agents service-side telemetry | **HIGH** ‚Äî No portal-native agent monitoring; must build custom dashboards |
| **Custom FoundryTracer events** | Our `trace_model_invocation`, `trace_decision`, `trace_tool_call` work regardless | Still work ‚Äî they're our code, not Foundry's | **NONE** ‚Äî Our custom telemetry layer is transport-agnostic |

### Key Finding

**`AIProjectInstrumentor` DOES instrument direct Responses API calls**, not just Foundry Agents. It hooks into the `AIProjectClient` HTTP pipeline. So basic OpenTelemetry spans and Application Insights integration survive the switch. What we lose is the **agent-semantic layer** ‚Äî spans that know about agent names, sessions, and tool orchestration steps. The Foundry portal's agent-specific monitoring views depend on this semantic layer.

### Mitigation Cost

To replicate agent-level observability on direct API:
- Add custom OTel spans wrapping each `responses.create()` call with agent name, session ID, and tool context as span attributes (~2 days of work).
- Build custom Application Insights dashboards or Kusto queries to replace the Foundry portal agent dashboard (~1 week).

## 2. Session & Thread Management

### What We Currently Have

Our session architecture spans three layers (documented in [session_manager.py](../../lib/src/holiday_peak_lib/agents/memory/session_manager.py)):

1. **Foundry server-side sessions** ‚Äî `AgentSession` with `service_session_id`. The Foundry Agents service maintains conversation history server-side. We pass `session` to `agent.run()` and get back `response.session` with the updated state.
2. **Redis (hot)** ‚Äî Compact `SessionSummary` (keywords, message count, last topic) for fast continuity decisions.
3. **Cosmos DB (warm)** ‚Äî Full session state including `_foundry_session_state` dict for Foundry thread restoration.

The decision flow in `evaluate_session_continuity()` uses keyword overlap analysis to decide whether to continue an existing Foundry thread or start fresh.

### What We Lose

| Session Capability | Foundry Agents | Direct API | Impact |
|---|---|---|---|
| **Server-side conversation history** | Foundry maintains full message history per session | **Lost** ‚Äî must pass full message history on every request | **HIGH** ‚Äî Increases payload size and token consumption for multi-turn conversations |
| **`service_session_id` assignment** | Foundry assigns and manages session IDs | Must generate and manage our own | **MEDIUM** ‚Äî Straightforward to implement but must handle ID uniqueness and lifecycle |
| **Thread continuation across restarts** | Session survives pod restarts (server-side state) | Requires full message replay from Cosmos | **MEDIUM** ‚Äî We already store full state in Cosmos, but token cost increases |
| **Smart continuity decisions** | Our `evaluate_session_continuity` works with Foundry's session model | Must adapt to carry full message context instead of session IDs | **MEDIUM** ‚Äî Architectural change to session_manager.py |

### Replication Strategy

We already have 80% of the infrastructure. The `session_manager.py` stores full message history in Cosmos. The change would be:
1. Instead of passing `AgentSession(session_id=X)` to Foundry, pass the full reconstructed message list to `responses.create()`.
2. Each request includes the conversation prefix (system + prior messages) + new user message.
3. Token cost increases proportionally to conversation length.
4. Implement a sliding-window or summarization strategy to cap context size.

**Estimated effort**: 3‚Äì5 days to refactor session management.

## 3. Agent Versioning & Drift Detection

### What We Currently Have

Our [foundry.py](../../lib/src/holiday_peak_lib/agents/foundry.py) implements a sophisticated versioning system:

1. **`_create_agent_version()`** ‚Äî Creates immutable agent versions via `agents_client.create_version()` using `PromptAgentDefinition` with model, instructions, reasoning, temperature, and top_p.
2. **`_check_instruction_drift()`** ‚Äî Compares local prompt instructions (SHA-256 hash) against the latest remote version. If different, automatically creates a new version.
3. **`_get_latest_version_instructions()`** ‚Äî Fetches remote instructions for drift comparison.
4. **Fallback instruction guard** ‚Äî `is_fallback_instructions()` check prevents publishing placeholder prompts to Foundry (Track A3 safety).

### What We Lose

| Versioning Capability | Foundry Agents | Direct API | Impact |
|---|---|---|---|
| **Immutable agent versions** | `create_version` creates timestamped, auditable versions | **Lost entirely** ‚Äî no concept of "agent version" in Responses API | **HIGH** ‚Äî Lose audit trail of prompt changes |
| **Instruction drift detection** | Compare local vs remote instructions automatically | **Lost** ‚Äî no remote source of truth for instructions | **HIGH** ‚Äî Must implement our own version control |
| **One-click rollback** | Restore previous version via Foundry portal or API | **Lost** ‚Äî must implement git-based rollback | **MEDIUM** ‚Äî Can use git tags + deployment pipeline |
| **Foundry portal version history** | Visual version timeline with diff view | **Lost entirely** | **MEDIUM** ‚Äî Developer convenience feature |
| **Fallback guard (Track A3)** | Prevents bad prompts from reaching production Foundry agent | Still needed ‚Äî reimplement against local version store | **LOW** ‚Äî Logic is in our code, not Foundry's |

### Replication Strategy

1. Store prompt versions in Cosmos DB or a dedicated git-tracked registry.
2. Each deployment creates a version record: `{version_id, sha256, instructions, model, reasoning, timestamp}`.
3. Drift detection compares deployed version hash against repo HEAD.
4. Rollback = redeploy previous container image (we already have this via AKS + GitHub Actions).

**Estimated effort**: 1 week to build a self-hosted version registry. Alternatively, continue using `ensure_foundry_agent` for version management while invoking models directly (hybrid approach).

## 4. Reasoning Control

### What We Currently Have

Our `_create_agent_version()` passes `reasoning` to `PromptAgentDefinition`:

```python
definition_kwargs["reasoning"] = reasoning  # e.g., {"effort": "minimal"}
```

The lifecycle manager in [foundry_lifecycle.py](../../lib/src/holiday_peak_lib/app_factory_components/foundry_lifecycle.py) configures per-role defaults:
- **Fast (SLM)**: `reasoning_effort = "minimal"` (env: `FOUNDRY_REASONING_EFFORT_FAST`)
- **Rich (LLM)**: `reasoning_effort = "low"` (env: `FOUNDRY_REASONING_EFFORT_RICH`)

There's also an auto-correction guard for gpt-5-nano/mini that floor `"none"` to `"minimal"`.

**Important code comment** in `FoundryAgentInvoker._request_response_impl`:
```python
# NOTE: reasoning_effort is an agent-DEFINITION parameter set at
# creation/version time via PromptAgentDefinition(reasoning=...).
# Runtime options.reasoning returns 400 for Foundry Agents.
```

### Direct API Comparison

| Parameter | Foundry Agents | Direct Responses API | Impact |
|---|---|---|---|
| **`reasoning_effort`** | Set at agent definition level; immutable per version | **Available as a runtime parameter** on `responses.create(reasoning={"effort": "..."})` | **POSITIVE** ‚Äî More flexible; can vary per request |
| **`temperature`** | Set at definition level via `PromptAgentDefinition` | Available as runtime parameter on each call | **POSITIVE** ‚Äî More flexible |
| **`top_p`** | Set at definition level | Available as runtime parameter | **POSITIVE** ‚Äî More flexible |
| **`max_output_tokens`** | Runtime via `ChatOptions(max_output_tokens=...)` | Runtime parameter on each call | **NEUTRAL** ‚Äî Same capability |

### Key Finding

**This is actually a win for direct API.** Foundry Agents lock reasoning_effort at agent-version creation time. If you want different reasoning_effort for the same agent (e.g., "minimal" for simple queries, "medium" for complex ones), you need separate agent versions. The Responses API allows per-request control, which aligns better with our SLM-first routing strategy where complexity determines effort level.

## 5. Tool Execution

### What We Currently Have

Our `FoundryAgentInvoker._prepare_invocation()` handles two tool paradigms:

1. **Callable tools** ‚Äî Python functions passed as `tools_raw`. These are forwarded to MAF's `FunctionInvocationLayer`, which handles the tool-call loop (model requests tool ‚Üí MAF executes ‚Üí model receives result ‚Üí continues).
2. **Schema-based tools** ‚Äî JSON schemas describing tools that aren't locally executable. These are injected into the system message as a prompt, and tool calls are extracted from the model's text response via `_extract_tool_calls_from_text()`.

### What We Lose

| Tool Capability | Foundry Agents (MAF) | Direct API | Impact |
|---|---|---|---|
| **Automatic tool-call loop** | MAF `FunctionInvocationLayer` handles multi-turn tool calls (model‚Üítool‚Üímodel‚Üítool‚Üíresponse) | Must implement the loop manually | **HIGH** ‚Äî Core orchestration feature; 50-100 lines of code to replicate |
| **Tool choice control** | `tool_choice` parameter at agent definition level | Available as runtime parameter on `responses.create()` | **POSITIVE** ‚Äî More flexible per-request |
| **Server-side tool execution** | MAF can execute tools server-side in the agent pipeline | Not available ‚Äî all tool execution is client-side | **LOW** ‚Äî We already run tools client-side via callable injection |
| **Schema-tool injection** | Our custom `_inject_tool_prompt` fallback for non-callable tools | Same approach needed | **NEUTRAL** ‚Äî Our code, not MAF's |

### Replication Strategy

The Responses API supports native function calling. The tool-call loop implementation:

```python
while True:
    response = await client.responses.create(messages=messages, tools=tools)
    if not response.tool_calls:
        break
    for tool_call in response.tool_calls:
        result = await execute_tool(tool_call)
        messages.append(tool_result_message(tool_call, result))
```

**Estimated effort**: 2‚Äì3 days. The Responses API's function calling is actually more straightforward than MAF's middleware-based approach.

## 6. Middleware & Extensibility

### What We Currently Have

MAF provides a middleware pipeline (`agent_framework.BaseAgent`):
- **Agent middleware** ‚Äî Pre/post-processing around the full agent run
- **Function middleware** ‚Äî Wraps individual tool executions
- **Chat middleware** ‚Äî Wraps model API calls

Our codebase uses MAF middleware indirectly through:
- `FoundryProviderPolicyStrategy` ‚Äî Sanitizes messages (strips system messages when `enforce_foundry_prompt_governance=True`)
- `GuardrailMiddleware` ‚Äî Enrichment validation, attribution, audit
- Correlation middleware ‚Äî Injects `correlation_id` into requests

### What We Lose

| Middleware Capability | Foundry Agents (MAF) | Direct API | Impact |
|---|---|---|---|
| **MAF middleware pipeline** | Structured pre/post hooks for agent, function, and chat | Must implement our own interceptor pattern | **MEDIUM** ‚Äî We barely use MAF's middleware directly; our guardrails are application-level |
| **Compaction strategies** | MAF supports conversation compaction for long threads | Must implement ourselves | **MEDIUM** ‚Äî Important for multi-turn sessions |
| **Custom tokenizers** | MAF allows custom tokenizer injection | Not applicable ‚Äî tokenization is server-side | **LOW** ‚Äî We don't use custom tokenizers |
| **Message serialization** | MAF `Message` ‚Üí content normalization | Direct API uses standard message format | **POSITIVE** ‚Äî Simpler; no `MAFMessage` wrapping |

### Key Finding

Our actual middleware usage is thin. The `GuardrailMiddleware`, `FoundryProviderPolicyStrategy`, and correlation middleware are all **our code**, not MAF framework features. They wrap `invoke_model()` at the `BaseRetailAgent` level, not at the MAF pipeline level. Switching to direct API would not break any of these.

**Estimated effort**: 1‚Äì2 days to remove MAF `Message`/`AgentSession` serialization and use standard OpenAI message format.

## 7. Content Safety & RAI

### What We Currently Have

`PromptAgentDefinition` supports a `rai_config` parameter for configuring Responsible AI guardrails. **However, our codebase does NOT currently use `rai_config`** ‚Äî a grep across the entire `lib/src` directory found zero references to `rai_config`, `content_filter`, `content_safety`, or `responsible_ai`.

Our content safety is handled at the application level via:
- `_FOUNDRY_HARDENING_BLOCK` in [prompt_loader.py](../../lib/src/holiday_peak_lib/agents/prompt_loader.py) ‚Äî Injects security directives into every agent's system prompt
- `FoundryProviderPolicyStrategy` ‚Äî Strips system messages when governance is enforced
- `GuardrailMiddleware` ‚Äî Validates enrichment requests and enforces attribution

### What We Lose

| RAI Capability | Foundry Agents | Direct API | Impact |
|---|---|---|---|
| **`rai_config` on agent definition** | Available but **unused** in our codebase | Not available as a parameter | **NONE** ‚Äî We don't use it |
| **Azure Content Safety integration** | Automatic when configured on the agent | Available on Azure OpenAI model deployments (deployment-level content filters) | **NONE** ‚Äî Content filtering is configured at the model deployment level, not the agent level |
| **Prompt injection protection** | Our `_FOUNDRY_HARDENING_BLOCK` handles this | Same ‚Äî it's our prompt, not Foundry's feature | **NONE** |

### Key Finding

**No impact.** Content safety in Azure AI is primarily enforced at the **model deployment level** (Azure OpenAI content filtering policies), not at the Foundry Agents layer. Our RAI strategy is prompt-based and application-level. Direct API calls to the same model deployment get the same content filters.

## 8. Cost Implications

### Foundry Agents Billing Model

As of April 2026:
- **Foundry Agents service**: No additional per-invocation charge beyond the underlying model tokens. The Agents service is included in the Azure AI Foundry project at no extra cost.
- **Token consumption**: Foundry Agents add overhead tokens for session management, tool orchestration, and system prompts. The overhead varies:
  - Simple single-turn: ~200‚Äì500 extra tokens (session metadata, agent instructions)
  - Multi-turn with tools: ~1,000‚Äì3,000 extra tokens (conversation history replay, tool schemas)
- **Storage**: Foundry stores session/thread state server-side. This is included but contributes to project storage costs.

### Direct API Cost Comparison

| Cost Factor | Foundry Agents | Direct API | Impact |
|---|---|---|---|
| **Per-invocation surcharge** | None | None | **NEUTRAL** |
| **Token overhead from agent layer** | ~200‚Äì3,000 extra tokens per call | None ‚Äî you control exactly what's sent | **POSITIVE** ‚Äî 5‚Äì15% token savings depending on conversation complexity |
| **Session storage** | Included in Foundry | Must pay for Redis + Cosmos (we already do) | **NEUTRAL** ‚Äî We already self-manage session storage |
| **Token cost for conversation replay** | Server-side; not billed again | Must re-send full conversation on each turn | **NEGATIVE** ‚Äî Multi-turn conversations cost more tokens on direct API |

### Net Cost Analysis

For **single-turn requests** (majority of our retail agents): Direct API is **5‚Äì15% cheaper** in token consumption.

For **multi-turn conversations** (cart intelligence, support): Direct API may be **10‚Äì30% more expensive** due to conversation replay, unless we implement summarization.

## 9. Future Roadmap Risk

### Microsoft's Investment Direction

Based on publicly available signals as of April 2026:

| Signal | Direction |
|---|---|
| Foundry Agents SDK reaching GA (`azure-ai-projects>=2.0.0`) | Heavy investment in Agents abstraction |
| Introduction of `PromptAgentDefinition` and versioning API | Treating agents as first-class managed resources |
| Responses API becoming the primary inference surface | Parallel investment in direct API |
| MAF SDK (`agent-framework`) active development | Framework investment continues |
| Azure AI Foundry portal agent dashboard improvements | Portal tooling tied to Agents |

### Risk Assessment

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| Microsoft deprecates direct API in favor of Agents-only | **Very Low** ‚Äî Responses API is the universal inference surface | Critical | Monitor Azure roadmap announcements |
| Microsoft adds agent-only features (multi-agent orchestration, built-in RAG, advanced tool routing) | **High** ‚Äî This is the value proposition of the Agents layer | Medium | Hybrid architecture preserves option to adopt |
| Direct API latency gap narrows (Foundry Agents optimization) | **Medium** ‚Äî Microsoft is aware of the latency overhead | Medium | Re-evaluate quarterly |
| Evaluation framework becomes Agents-only | **Medium** ‚Äî Foundry evaluators currently work with any response | High | Ensure eval pipeline stays decoupled |

### Key Finding

**The safest bet is the hybrid approach.** Microsoft is investing heavily in both paths. The Responses API is not going away (it's the foundation of everything). Foundry Agents will gain exclusive features over time, particularly around multi-agent orchestration, built-in memory, and managed tool ecosystems. Keeping a foot in both worlds preserves future options.

### Implementation Sketch for `DirectResponsesInvoker`

```python
class DirectResponsesInvoker:
    """Direct Responses API invoker ‚Äî no Foundry Agents layer."""

    def __init__(self, config: FoundryAgentConfig, *, timeout: float | None = None):
        self.config = config
        self._client: AIProjectClient | None = None
        self._timeout = timeout or 30.0  # Much lower ‚Äî no agent overhead

    def _ensure_client(self) -> AIProjectClient:
        if self._client is None:
            credential = self.config.credential or DefaultAzureCredential()
            self._client = AIProjectClient(
                endpoint=self.config.endpoint, credential=credential
            )
        return self._client

    async def __call__(self, **kwargs) -> dict[str, Any]:
        client = self._ensure_client()
        messages = kwargs.pop("messages", [])
        model = kwargs.pop("model", self.config.deployment_name)
        temperature = kwargs.pop("temperature", 0.2)
        reasoning_effort = kwargs.pop("reasoning_effort", "minimal")

        response = await asyncio.wait_for(
            client.responses.create(
                model=model,
                input=messages,
                temperature=temperature,
                reasoning={"effort": reasoning_effort},
            ),
            timeout=self._timeout,
        )
        # ... normalize response to our standard dict format
```

This `DirectResponsesInvoker` plugs into `ModelTarget.invoker` identically to `FoundryAgentInvoker`. No changes needed to `BaseRetailAgent`, `invoke_model()`, or the routing layer.

## Final Recommendation

### Decision: Adopt Hybrid Architecture

**Phase 1 (Weeks 1‚Äì3)**: Implement `DirectResponsesInvoker`
- Create a new invoker class that implements the same `ModelInvoker` interface
- Wire it into `ModelTarget` alongside `FoundryAgentInvoker`
- Add a per-service configuration flag: `AGENT_INVOCATION_MODE=direct|foundry`
- Migrate 12‚Äì14 latency-critical, single-turn agents to direct API
- Expected latency improvement: 2‚Äì5s per request on migrated agents

**Phase 2 (Weeks 3‚Äì5)**: Telemetry and session parity
- Add custom OTel spans for direct API calls with agent-semantic attributes
- Build Application Insights dashboards for direct API agents
- Implement client-side tool-call loop for any direct-API agents needing tools
- Implement conversation summarization for the few multi-turn direct-API agents

**Phase 3 (Ongoing)**: Evaluate and optimize
- Monitor latency, cost, and quality metrics across both paths
- Move agents between modes based on data
- Evaluate Foundry Agents latency improvements in future SDK releases
- Preserve the option to consolidate on a single path when the trade-offs shift

### Trade-Off Summary

| Factor | Weight | Verdict |
|---|---|---|
| Latency reduction (2‚Äì5s) | **Critical** ‚Äî user-facing retail | Strongly favors direct API for simple agents |
| Telemetry preservation | **High** | Achievable with moderate effort |
| Session management complexity | **Medium** | Already partially built; direct API adds token cost |
| Versioning / drift detection loss | **Medium** | Can use git + Cosmos-based registry |
| Future feature access | **Medium** | Hybrid preserves optionality |
| Implementation effort | **Medium** | ~3‚Äì5 weeks for full hybrid with parity |
| Operational complexity | **Low‚ÄìMedium** | Two invocation paths to maintain |

The hybrid approach delivers the latency win where it matters most (65%+ of our agents are single-turn) while preserving Foundry Agents' managed session state, versioning, and tool orchestration for the complex workflows that genuinely benefit from them.
